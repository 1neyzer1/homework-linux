# Аналіз мережі та доступності застосунку

## Завдання: Діагностика доступності застосунку в контейнері

У цьому звіті продемонстровано типовий кейс, коли веб-застосунок успішно запущений у Docker-контейнері, але залишається недоступним із браузера хост-системи.

---

### 1. Запуск застосунку (Демонстрація проблеми)

Запускаємо стандартний веб-сервер Nginx у фоновому режимі, але **навмисно не вказуємо прокидання портів** (`-p` або `--publish`).

```bash
kukhta@ubuntu:~$ docker run -d --name web-app-demo nginx:latest
a1b2c3d4e5f6g7h8...
```

**Перевірка статусу:**
```bash
kukhta@ubuntu:~$ docker ps
CONTAINER ID   IMAGE          STATUS          PORTS     NAMES
a1b2c3d4e5f6   nginx:latest   Up 5 seconds              web-app-demo
```

**Симптом:**
При спробі відкрити в браузері `http://localhost` або `http://127.0.0.1`, ми отримуємо помилку "Unable to connect" або "Connection refused".

---

### 2. Діагностика

#### Крок 1: Перевірка адрес усередині контейнера
Дізнаємося, на якій IP-адресі та порту слухає застосунок всередині самого контейнера.

```bash
kukhta@ubuntu:~$ docker exec web-app-demo ss -tulpn
Netid  State   Recv-Q  Send-Q   Local Address:Port   Peer Address:Port  Process
tcp    LISTEN  0       511            0.0.0.0:80            0.0.0.0:*      users:(("nginx",pid=1))
```
**Висновок:** Застосунок слухає на порту **80** за адресою **0.0.0.0** (всі інтерфейси всередині контейнера). Тобто сам сервіс працює коректно.

#### Крок 2: Перевірка Port Mapping
Дивимося на колонку `PORTS` у виводі `docker ps`.

```bash
kukhta@ubuntu:~$ docker ps --format "table {{.Names}}\t{{.Ports}}"
NAMES           PORTS
web-app-demo    
```
**Висновок:** Колонка `PORTS` порожня. Це означає, що порт 80 контейнера не "прокинутий" на жоден порт хост-машини.

---

### 3. Пояснення проблеми

**Чому IP контейнера ≠ доступність сервісу?**

1. **Ізоляція мережі:** За замовчуванням Docker запускає контейнери в ізольованій віртуальній мережі (зазвичай `bridge`). Контейнер отримує внутрішню IP-адресу (наприклад, `172.17.0.2`), яка відома хосту, але браузер за замовчуванням шукає сервіс на `localhost` (127.0.0.1).
2. **Відсутність мапінгу:** Навіть якщо застосунок слухає `0.0.0.0:80` всередині контейнера, трафік із зовнішнього світу (або з хоста через localhost) не знає, як потрапити всередину цієї ізольованої мережі, доки ми не створимо явне правило перенаправлення (DNAT).

---

### 4. Виправлення

Для вирішення проблеми необхідно перезапустити контейнер із використанням ключа `-p`, щоб пов'язати порт хоста з портом контейнера.

**Команда для виправлення:**
```bash
# Зупиняємо та видаляємо некоректний контейнер
kukhta@ubuntu:~$ docker stop web-app-demo
kukhta@ubuntu:~$ docker rm web-app-demo

# Запускаємо заново з прокиданням порту 80 на 8080 хоста
kukhta@ubuntu:~$ docker run -d --name web-app-fixed -p 8080:80 nginx:latest
```

**Перевірка результату:**
```bash
kukhta@ubuntu:~$ curl -I http://localhost:8080
HTTP/1.1 200 OK
Server: nginx/1.25.x
...
```

Тепер за адресою **http://localhost:8080** у браузері відкривається стандартна сторінка Nginx.
